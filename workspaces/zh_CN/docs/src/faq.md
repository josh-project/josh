# 常见问题

## Josh 与 git sparse-checkout 有何不同？

Josh 基于 git 对象图，与在文件系统上检出文件和工作树（即 sparse-checkout 的关注点）无关。稀疏检出不会改变对象数据库的内容，也不会影响通过网络下载的内容。两者在需要时可以一起使用。

## Josh 与部分克隆有何不同？

部分克隆（partial clone）会根据条件只下载对象数据库的一部分。它仍然是同一个对象数据库：历史包含相同的提交及相同的 SHA-1 标识，并且允许以后再加载此前被跳过的对象。Josh 则会创建一个不引用被跳过部分的替代历史，在本质上类似于 git filter-branch，但更快、功能更多、且提供不同的交互方式。

## Josh 与子模块有何不同？

git 子模块（submodules）是多个独立仓库，通过引用特定提交的 SHA 来相互关联。Josh 支持 monorepo（单体仓库）方式：所有代码保存在同一仓库中，便于同步。Josh 可以将任意子文件夹（或工作区下更复杂的文件夹组合）视为独立的 git 仓库，与主 monorepo 之间实现双向透明的同步。因此 Josh 的功能超出子模块，且使用上更简单、更快捷。

## Josh 与 `git subtree` 有何不同？

Josh 的基本思想与 `git subtree` 相似。然而与 `git filter-branch` 一样，`git subtree` 在常规使用中通常过慢，即便是中等规模的仓库也如此。`git subtree` 只有在压缩提交并因此丢失历史时才可能获得可接受的性能。Josh 的核心本质上是对 `git subtree split` 的高性能实现，针对在同一仓库内频繁运行进行了专门优化。

## Josh 与 `git filter-repo` 有何不同？

`josh-filter` 和 `git filter-repo` 都能快速重写 Git 历史，在简单场景下可实现相同目的。

在更复杂的用例中，应根据目标选择工具：`git filter-repo` 在过滤能力上更灵活、可选项更多，例如支持重写提交信息或插入任意脚本作为过滤步骤。

Josh 使用 DSL（领域特定语言）而非任意脚本，因此在可实现的过滤类型上更受限制；但它支持增量过滤和双向操作，能够在原始仓库与过滤后仓库之间相互转换变更。

## Josh 与以上所有替代方案有何不同？

Josh 包含 `josh-proxy`，以服务形式提供仓库过滤，主要用于支持 monorepo 工作流。
