### New features

* **Honor `change-id` commit header from jj / GitButler** (#2118)
  Jujutsu and GitButler store their stable change identifier as a custom
  `change-id` header on the git commit object itself, not as a message-body
  trailer. josh-changes now reads that header (and lets it override a
  conflicting `Change:` trailer), so jj users no longer need to maintain a
  parallel `Change:` trailer to use stack publish.

* **Path-intersection downstack algorithm** (#2022)
  The old downstack algorithm walked intermediates forward and pulled in every
  commit until the change's diff applied cleanly, which included unrelated
  commits that simply happened to sit earlier in the stack. The new algorithm
  does a single backward scan, marking only intermediates whose touched paths
  intersect the change's affected-path set. Unrelated intermediates are now
  correctly skipped regardless of their position in the stack.

* **Add `--version` flag to `josh-proxy`** (#2120)
  `josh-proxy` already exposed a `/version` HTTP endpoint but had no CLI flag.
  Operators inspecting a deployed binary on disk can now run
  `josh-proxy --version` (or `-V`) without starting the server.

* **Support git revision syntax in input refs** (#2034)
  `josh-filter` now accepts full git revision expressions (e.g. `HEAD~1`)
  wherever an input ref is expected, not just short reference names.

* **Preview PR actions in `josh changes publish --dry-run`** (#2134)
  `josh changes publish --dry-run` now prints the pull-request actions that
  would be taken (which PRs would be created and which existing PRs would be
  updated, including base-branch and draft-state changes), so you can
  preview the GitHub side effects without performing the publish.

* **Simplified Chinese (`zh_CN`) documentation** (#2126)
  The documentation site now ships a Chinese translation, with a language
  picker and an outdated-translation banner. Translation tooling
  (`mdbook-gettext`, a `build-docs.sh` helper, and the `en_001` / `zh_CN`
  locale codes) is wired into the docs build.

### Breaking changes

* **Rename `josh publish` to `josh changes publish`** (#2023)
  The `publish` subcommand has moved under a new `changes` subcommand group,
  making room for further changes-related subcommands alongside `publish`.

* **Drop "stack" push mode; rename proxy magic ref `refs/split/...` to `refs/publish/...`** (#1981)
  The old "stack" push mode has been removed. The proxy's publish magic ref
  has also been renamed from `refs/split/for/<branch>` to
  `refs/publish/for/<branch>` so that the proxy and the CLI use the same
  vocabulary.

* **Replace `Requires:` trailer with `Change-Series:`** (#2022)
  The downstack algorithm has been rewritten (see above). As part of that, the
  `Requires:` commit footer used to force ordering between commits has been
  replaced with `Change-Series:`. Assign two or more commits to a named series
  to keep them merged in order even when they touch disjoint files.

* **Change footers are only parsed in the trailing footer block** (#2080)
  `Change:`, `Change-Id:`, and `Change-Series:` lines are now only recognised
  when they appear in the trailing block of trailers at the end of the commit
  message. A body line that happens to start with one of those prefixes is no
  longer mis-parsed as a real trailer.

### Bug fixes

* **Downstack now handles binary files** (#2010)
  The previous diff-based downstack approach did not work with binary files.
  Downstack now uses a 3-way merge, which handles binary files correctly.

* **Filter flattening no longer absorbs non-invertible filters into a compose** (#2029)
  The flatten optimiser step would absorb non-invertible filters (such as
  `:+st/a`) into a compose, even though compose requires its elements to be
  invertible — causing later application to fail. The optimiser now leaves
  such filters in place.

* **Filtered `Cargo.toml` strips dual `path` + `version` declarations** (#2109)
  In `[workspace.dependencies]`, `josh-*` crates are declared with both a
  `path` and a `version`. In a filtered worktree without `.git`, cargo would
  sometimes resolve these against crates.io despite the `path` field. A new
  `ws/patched-cargo.josh` filter rewrites these entries inside the build
  container; the on-disk `Cargo.toml` is untouched.

* **Fix shutdown on SIGINT** (#2012)
  `josh-proxy` now shuts down cleanly when interrupted.

* **Fix `josh` CLI invocation from git worktrees** (#2133)
  The CLI now works correctly when invoked from a linked git worktree.

### Performance

* **Bundle publish/push refs into a single `git push`** (#2142)
  Push and publish now send all computed refs in a single `git push`
  invocation rather than spawning one process per ref. This also makes
  `--atomic` meaningful across the entire set of refs rather than only
  applying to one ref at a time.

* **Cache history graph root reachability data** (#2006)
  Reachability checks during history filtering are now backed by a cached
  data structure, giving much faster "are these tips reachable from each
  other" answers when only existence (not the nearest common ancestor)
  matters.

* **Store known filters in the cache** (#1976)
  Known filters are now persisted in the cache, enabling rebuilds of filter
  state across restarts.

### Experimental

These changes affect features that are opt-in via
`JOSH_EXPERIMENTAL_FEATURES=1`. Experimental features make no stability
promises, so the changes below are not considered breaking.

* **Rename object-dereference filter `:*path` to `:#path`** (#1917)
  The experimental object-dereference filter operator has been renamed from
  `:*` to `:#`. Filter expressions using `:*path` (or `:*/path`) must be
  updated to `:#path` (or `:#/path`). The corresponding documentation in
  `docs/src/reference/experimental.md` has been updated.

* **Starlark filters see objects written via the mempack backend** (#1970)
  Starlark evaluation previously opened a fresh repository handle, so
  objects written through the original handle were missing when that handle
  used the mempack backend — causing filter evaluation to fail. Starlark
  now runs against the active transaction's repository handle and sees
  those objects.
